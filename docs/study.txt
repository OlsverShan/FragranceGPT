FragranceGPT 项目全面解析

  ---
  一、项目概述

  这是你用 LLM 技术栈解决香水领域专业问题 的个人项目，全称 FragranceGPT。核心任务：给定 5 个主香调（如 citrus, woody,
  aromatic），让 AI 预测香水的前调/中调/后调具体原料，并推荐相似的高评分真实香水。

  数据来自 Kaggle 上 Fragrantica.com 的 24,063 款香水，包含 84 种香调、1,655 种香料原料、用户评分等。

  三、项目包含的 8 大技术模块

  1. 数据引擎 (src/data.py)

  - 加载 24,063 条香水数据，解析欧洲数字格式
  - 清洗香调名称（去除修饰词、提取核心成分）
  - 构建前/中/后调结构化数据集

  2. 向量检索引擎 RAG (src/rag.py)

  - ChromaDB 持久化向量存储，24K 香水嵌入
  - sentence-transformers (all-MiniLM-L6-v2) 将每款香水编码为 384 维向量
  - 余弦相似度检索 Top-5 相似香水作为 LLM 提示参考
  - 这是项目中效果最好的单一技术（F1 +20.5%）

  3. 消融实验（7 种方法对比）

  你设计了严谨的科学实验，每种方法都定量对比：

  ┌────────────────┬─────────┬───────────────┐
  │      方法      │ 模糊 F1 │     结论      │
  ├────────────────┼─────────┼───────────────┤
  │ 随机猜测       │ 0.006   │ 理论下界      │
  ├────────────────┼─────────┼───────────────┤
  │ 频率统计       │ 0.327   │ 纯统计基线    │
  ├────────────────┼─────────┼───────────────┤
  │ LLM 零样本     │ 0.393   │ 裸调基线      │
  ├────────────────┼─────────┼───────────────┤
  │ LLM + Few-shot │ 0.360   │ 单独有效      │
  ├────────────────┼─────────┼───────────────┤
  │ LLM + RAG      │ 0.443   │ 最佳方案      │
  ├────────────────┼─────────┼───────────────┤
  │ RAG + Few-shot │ 0.442   │ 冗余（-0.2%） │
  ├────────────────┼─────────┼───────────────┤
  │ 多 Agent 协作  │ 0.418   │ 有害（-5.6%） │
  └────────────────┴─────────┴───────────────┘

  4. 模糊匹配系统 (src/synonyms.py)

  解决 "cedarwood" ≠ "cedar" 这类同义词被误判为错误的问题：
  - 第1层：120 条人工同义词映射
  - 第2层：LLM 构建 1,650 条同义词表
  - 第3层：规则规范化（去地理前缀、颜色修饰词）
  - 兜底：Levenshtein 编辑距离（阈值=2）

  5. LLM-as-Judge 评估 (src/evaluation.py)

  除 F1 外，用另一个 LLM 对预测结果从 4 个维度打分：
  - 关键发现：你的预测得分 4.37/5，高于 Fragrantica 真实数据 3.82/5
  - F1 与 Judge 评分相关系数 = -0.139（负相关），证明 F1 衡量记忆，Judge 衡量专业度

  6. 微调 Qwen2.5-7B (buildtest/finetune/)

  - 使用 Unsloth + QLoRA 4-bit 量化微调
  - 训练 22,859 样本，3 epochs，RTX 5080 上训练 1.5 小时
  - 可训练参数仅 40M / 7.6B (0.53%)
  - 结果：模糊 F1 = 0.418，完全本地免费运行

  7. 评分预测系统 (src/rating_predictor.py + src/feature_enrichment.py)

  - XGBoost 模型：TF-IDF (500维) + 品牌/调香师编码 + 年份 + 性别 + 香调多热编码 + LLM 特征增强
  - 3-Agent 特征增强管线：Fame Collector (LLM知识) + Data Miner (纯统计) + Synthesis Director (交叉验证)
  - 性能：MAE = 0.17，Pearson r = 0.54

  8. 8 人设评审团 (src/multi_persona_rater.py)

  8 个有冲突审美的 AI
  角色（古典派、极简派、先锋派、商业总监、乌木大师、格拉斯匠人、市场研究、夸夸师）并行打分，分歧度本身就是信号。

  9. Fused 评分融合 (src/fused_rater.py)

  XGBoost（校准精度）+ 8人设LLM（审美信号）的加权融合，根据专家一致性动态调整权重。

  10. Web 应用 (app/streamlit_app.py + backend/)

  - Streamlit 前端，4 个标签页：香水推荐、评分预测、市场分析、历史记录
  - FastAPI + WebSocket 后端，异步任务管理
  - 中英文双语（84 个香调 + 300+ 原料的完整翻译映射）
  - 购买链接查询、香水顾问对话式推荐

  六、面试核心话术

  "为什么做这个项目？"

  ▎ 我在 DSM-Firmenich 做过香精微胶囊和口腔崩解片实习，发现感官评价高度依赖人的主观判断，没有标准化工具。LLM
  ▎ 擅长处理模糊的、基于自然语言的主观判断，所以我用 LLM 技术栈做了这个评香系统。

  "最核心的技术发现是什么？"

  ▎ RAG 是这个任务的主导技术——从 24K 香水数据库中动态检索比 LLM 的预训练记忆靠谱得多（+20.5%
  ▎ F1）。同时我证明了不是所有热门技术都适用——Few-shot 相对 RAG 是冗余的，LangGraph 多 Agent 编排反而有害（-14%）。

  "怎么评估系统好不好？"

  ▎ 双指标。F1 衡量词汇命中率，LLM-as-Judge 衡量专业质量。我发现了 F1 和 Judge 的负相关（-0.139），证明在主观领域，只看
  ▎ F1 是不够的——模型预测（4.37/5）甚至被裁判认为比众包真值（3.82/5）更好。

  "遇到什么困难？"

  ▎ 评估指标的修正。最初精确匹配 F1 很低，因为 "cedarwood" vs "cedar"
  ▎ 被当成不同答案。我设计了三层模糊匹配回退机制，所有方法的 F1 修正了 9-17%，而且不需要改任何模型代码。

  这个项目最大的优势是全国独一无二——面试官看 100 份"做了个 RAG 问答"后，看到你的香水评香系统会停下来认真看。加上你有 DSM
   的领域知识背书，这直接变成了你的护城河。


 第一轮：RAG 基础（必问，必须答好）

  Q1：「你说 RAG 是主导技术，给我讲一下你的 RAG 管线是怎么工作的？」

  你要答出的核心流程：

  ▎ 分三步。第一步离线建库：把 24,063 款香水每条转成一段文本 "Accords: citrus, woody | Notes: Top: bergamot... Middle:
  ▎ ... Base: ..."，用 sentence-transformers 的 all-MiniLM-L6-v2 模型编码成 384 维向量，存入
  ▎ ChromaDB。第二步在线检索：用户输入的 accords 同样编码成向量，在 ChromaDB 中用余弦相似度找 Top-5
  ▎ 最近邻。第三步增强生成：把检索到的 5 款香水的真实配方作为参考注入 Prompt，LLM 参考后输出结构化预测。

  追问：「为什么选 ChromaDB 而不是 FAISS 或 Pinecone？」

  ▎ ChromaDB 是嵌入式、零配置的，适合原型阶段，数据和应用一起部署。FAISS
  ▎ 是纯库不是数据库，需要自己管理持久化和元数据。Pinecone 是云服务，有网络延迟和成本。对于 24K 条数据这个量级，ChromaDB
  ▎  的 HNSW 索引完全够用。

  追问：「你说的 HNSW 是什么？为什么不用暴力搜索？」

  ▎ HNSW 是分层可导航小世界图，一种近似最近邻算法。它建多层图结构，上层稀疏做长距离跳跃，下层稠密做精细搜索。查询时从顶
  ▎ 层贪心下降到底层，时间复杂度 O(log N)。24K 数据用暴力搜索（O(N)）也行，但 HNSW
  ▎ 是大规模向量的标准做法。关键是它不需要训练，增量插入也方便。

  Q2：「你的 embedding 模型选的是 all-MiniLM-L6-v2，为什么？有什么缺点？」

  你要答出：

  ▎ 选它是因为轻量（80MB）、速度快（CPU 上毫秒级）、384
  ▎ 维存储成本低。缺点也很明显：它是通用英文模型，不是为香水领域训练的。"bergamot" 和 "citrus"
  ▎ 在通用语义空间里的关系，跟调香领域的专业关系可能不一样。升级方向是 BGE-large 或在香水语料上做 domain-adaptive
  ▎ pretraining 或对比微调。

  追问：「什么叫 domain-adaptive pretraining？具体怎么做？」

  ▎ 就是拿通用预训练模型，在香水领域的无标注文本上继续 MLM（掩码语言模型）训练，让模型适应香水词汇的分布和共现关系。具体
  ▎ 做法：收集香水评测文本、调香文献、原料说明，用这些文本对 MiniLM 做几轮 MLM 继续预训练。几百 MB 的领域文本就行，GPU
  ▎ 上跑几小时。

  ---
  第二轮：Embedding 与向量检索（往深了挖）

  Q3：「你说用余弦相似度，为什么不是欧氏距离？两者什么区别？」

  这是经典基础题，必须会：

  ▎ 余弦相似度衡量的是方向一致性，欧氏距离衡量的是绝对位置差异。向量 [1,2,3] 和 [2,4,6]
  ▎ 方向完全相同（余弦=1），但欧氏距离很大。在文本嵌入中，两个意思相同但长度不同的句子，方向应该一样——所以用余弦。HNSW
  ▎ 在 ChromaDB 中本身也支持余弦空间。

  追问：「如果向量没有归一化，余弦和欧氏距离有对应关系吗？」

  ▎ 有。对归一化向量（L2=1），$||A-B||_2^2 = 2 - 2\cos(A,B)$，两者完全等价。但 MiniLM
  ▎ 的输出不一定归一化了，所以存储时直接用余弦更安全。

  Q4：「向量维度为什么是 384？512 和 768 的模型你没试吗？维度越高越好吗？」

  ▎ 384 是 all-MiniLM-L6-v2 的输出维度。维度不是越高越好——高维有"维度灾难"问题，数据在高维空间中趋于等距，最近邻的区分度
  ▎ 反而下降。存储成本也线性增长（384维 × 24K × 4字节 ≈ 37MB，768维就是 74MB）。实践上 384 对 24K
  ▎ 级的语义检索已经足够区分度。

  ---
  第三轮：Prompt Engineering（你的核心工作）

  Q5：「你的 Prompt 里写了什么？为什么要 include exactly 5 specific notes per layer？」

  ▎ 核心设计：角色设定（professional perfumer）、任务说明、参考信息（RAG 检索结果）、输出格式约束（JSON with exactly 5
  ▎ notes per layer）。固定 5 个是为了公平对比——7 种方法都用同样的输出格式，F1
  ▎ 计算才有可比性。如果不固定数量，有的方法输出 3 个有的输出 10 个，Precision/Recall 没法比。

  追问：「那固定 5 个本身不是也有问题吗？真实香水每层平均就 3 个 notes。」

  你要承认这个局限：

  ▎ 确实有问题。固定 5 个人为拉低了 Precision（因为我们硬要模型猜 5 个，但真值往往只有 3 个），这也是为什么所有方法的
  ▎ Precision 都偏低。更好的做法是让模型学到数据中的自然分布然后输出不定数量。微调模型就做到了这一点——它从训练数据的
  ▎ note 数量分布中学到每层只输出约 3 个，所以微调模型的 Precision（0.293）比 RAG（0.256）高。

  Q6：「如果我让你不修代码，只改 Prompt 来提升效果，你会怎么改？」

  ▎ 三个方向。第一，加 chain-of-thought："先分析这 5 个 accords 的典型配方风格，再逐层列出 notes"。第二，在 Prompt 里加
  ▎ calibration 提示："你的预测应该基于调香学原理，而非简单复制参考资料中的频繁 note"，减少 LLM
  ▎ 盲目复制高频原料的倾向。第三，加
  ▎ constraints："前调应包含易挥发的柑橘/清新原料，后调应包含持久性原料如木质/麝香"，把领域知识编码进 Prompt。

  ---
  第四轮：评估指标（最能体现深度的部分）

  Q7：「F1 和 LLM-as-Judge 评分之间相关系数 -0.139，你怎么解读？」

  这是你项目最亮的点，必须讲透：

  ▎ 负相关说明两个指标衡量的是不同维度。F1 衡量的是"你能多精确地复现 Fragrantica 的众包标签"——本质是记忆测试。Judge
  ▎ 衡量的是"你的预测是否专业、合理、协调"——本质是泛化/推理测试。负相关说明：有些香水 Fragrantica 标签本身不专业（Judge
  ▎ 打分低），但模型如果死记硬背它们，F1 就高；反之，模型给出更专业但与 Fragrantica 不同的答案，F1 低但 Judge 高。

  追问：「你拿 LLM 当裁判，怎么保证 LLM 裁判本身没有 bias？」

  ▎ 不能完全保证，这是 LLM-as-Judge 的固有限制。我做了几个措施缓和：用了一个独立于预测模型的 API key 做裁判（虽然还是
  ▎ DeepSeek，但至少换了一个调用上下文）；裁判 Prompt 设了结构化的 4 个评分维度，降低主观性；用 30
  ▎ 个样本的均值做结论。但诚实地讲，LLM-as-Judge 仍然可能偏好更"标准化"的回答——这需要人类评香师的 ground truth
  ▎ 来最终验证。

  追问：「那你为什么不直接找人类评香师来验证？如果你真心想做严肃研究的话？」

  ▎ 说实话，资源的限制。找人类评香师需要预算、时间、招募，这不是一个个人项目能承担的范围。但我是有意识地选择了一个有客观
  ▎  Ground Truth（Fragrantica 标签）的领域来弥补——这样我先有了一个可自动化的评估基线，然后用 LLM-as-Judge
  ▎ 作为补充信号。

  Q8：「你的模糊匹配三层的设计，每层分别解决了什么不同的问题？为什么三层顺序不能换？」

  这是设计题，体现工程思维：

  ▎ 第一层硬编码是精确的（人工校验过），放在最前面因为速度最快且零歧义——如果 "cedarwood" 被规则误处理成 "cedarwood"
  ▎ 而不是 "cedar"，我们就损失了正确匹配。第二层 LLM 同义词表覆盖面广（151 个有变体的组），但要放在硬编码后面，因为 LLM
  ▎ 构建的表可能有错（比如 LLM 可能把 "sandalwood" 和 "sandal"
  ▎ 错误归到不同组）。第三层规则规范化最激进（直接截断前缀后缀），容易误伤，放在最后是安全网的角色。

  特点：前三层都不改原始语义，只是名称归一化。编辑距离是"最后手段"，因为它不知道 "white musk" 和 "musk"
  是不是同一个东西——它只看字符差异。把编辑距离放在规则之前会导致大量误匹配——比如 "white
  pepper"（EditDist=6）可能不会被匹配，而 "red musk" vs
  "musk"（EditDist=4）会被错误匹配。所以要放在最后，且阈值尽量小（阈值=2）。

  Q9：「你说 F1 和 Judge 负相关是因为 ground truth 有噪声。那你怎么定义'噪声'？众包数据难道就一定不可靠吗？」

  ▎ 不是不可靠，是高方差。Fragrantica 的数据是任何用户都能提交的，有些香水 notes 标注不全（缺了关键的香料），有些层级划
  ▎ 分有争议（某成分到底是中调还是后调）。不是所有数据都有噪声，但是噪声确实存在。我们 LLM-Judge
  ▎ 打分结果提供了一个佐证——当系统给出与 Fragrantica 不同但结构更标准化、专业配比更协调的预测时，Judge 给了高分。

  追问：「你这个说法有数据支持吗？你能给我看一个具体的例子吗？」

  ▎ 可以。比如评测时发现：对于某款柑橘木质调的香水，Fragrantica 把 "vanilla"
  ▎ 放在了前调——这在调香原理上几乎是不可能的，前调应该在 15 分钟内挥发完，vanilla 是高沸点、留香持久的基调原料。我们的
  ▎ RAG 把 vanilla 放在后调，从 F1 角度"错了"，但从专业角度是"对"的。Judge 在这类案例上给了我们更高的评分。

  ---
  第五轮：微调与模型（硬核部分）

  Q10：「你微调 Qwen2.5-7B 用了 QLoRA。解释一下 LoRA 的原理，以及 QLoRA 和 LoRA 的区别。」

  这是基础题，每个做微调的人都应该掌握：

  ▎ LoRA（Low-Rank Adaptation）的核心思路是：预训练大模型的权重矩阵 W 是满秩的，但微调时的参数更新 ΔW
  ▎ 其实是低秩的。所以把 ΔW 分解为两个小矩阵的乘积 A×B（A 是 d×r，B 是 r×k，r 一般取 8-64），只训练 A 和 B。推理时 W' =
  ▎ W + A×B，没有额外延迟。

  ▎ QLoRA 在 LoRA 基础上加了三个技术：4-bit NormalFloat
  ▎ 量化（一种对正态分布权重优化的量化方案）、双重量化（对量化常数再做一次量化），和分页优化器（用 CPU 内存当 GPU
  ▎ 显存的溢出区）。本质是：基础模型权重用 4-bit 存，LoRA 适配器用 BF16 训练，大幅降低显存需求——7B 模型从 14GB 降到约
  ▎ 5GB 显存。

  追问：「你的 r=16, alpha=16 什么意思？为什么 r 不取更大的？」

  ▎ r 是低秩分解的秩。r=16 意味着 ΔW 矩阵分解为 d×16 和 16×k 两个矩阵。alpha 是缩放因子，实际学习率会乘以 alpha/r。r
  ▎ 大了会更多可训练参数但可能过拟合，r 小了表达能力不够。r=16 在 7B 模型上是一个平衡点。我的训练数据显示效果和 RAG
  ▎ 有差距（0.418 vs 0.443），说明这个任务的难点不是模型容量的限制，而是缺少实时检索的外部知识——即使把 r=64
  ▎ 这也是解决不了的问题。

  Q11：「微调模型不如 RAG，你觉得根本原因是什么？」

  你要给出有洞察的回答：

  ▎ 根本原因是知识的外部性。这个任务不是"学习一个映射函数"那么简单——24K 个香水配方是一个不断增长的、分布高度非均匀的外部
  ▎ 知识库。微调模型试图把所有这些知识压缩进模型参数，但这样做效率很低，而且模型不会自动"知道"什么时候该输出哪个
  ▎ variant。RAG 的优势在于：查询时动态检索，根据当前 accords 精确找到参考对象，不需要预先记住 1655
  ▎ 种原料的所有共现模式。

  Q12：「你对 24K 条数据做了 80/20 split，但数据是有品牌和年份信息的——你这 split 有什么问题吗？」

  ▎ 这是一个好问题。纯粹的随机 split 可能会导致数据泄露——同一品牌的不同香水可能分散在 train 和 test
  ▎ 中，导致模型通过记住品牌来作弊。更严格的 split 应该按时间去（比如 train 用 2015 年前的香水，test 用 2015
  ▎ 年后的），或者是按品牌去 split。我目前的 split 方式对于回归任务来说是可以接受的（MAE
  ▎ 计算没有异方差问题），但如果要进一步研究，这确实是一个改进点。

  ---
  第六轮：XGBoost 与特征工程

  Q13：「你 XGBoost 模型的 MAE 是 0.17，Pearson r=0.54。这些数字什么水平？怎么判断它们好不好？」

  ▎ MAE=0.17 意味着平均预测偏差只有 0.17 分（满分 5 分）。相对于基线 MAE（大约 0.38，即全部猜测均值），我们的模型有 55%
  ▎ 的改善。但 r=0.54 说明模型只解释了不到 30% 的评分方差——评分预测本身是一个很难的任务，因为用户评分受到品牌效应、包装
  ▎ 设计、营销、社会影响等文本特征无法捕捉的因素影响。

  追问：「0.17 的 MAE 是不是说明你的模型其实不行，因为评分范围是 1-5？」

  ▎ 要看 relative 的表现，不是只看绝对 MAE。评分范围虽然是 1-5，但实际评分集中在 3.5-4.5
  ▎ 之间（高评分的流行趋势），所以基线 MAE（全部猜均值）其实就是约 0.38。55% 的提升是显著的。而且 XGBoost
  ▎ 在这个项目中是作为校准信号使用的，不是独立的评分工具。

  Q14：「你用了品牌和调香师的 target encoding——解释一下 target encoding 的 leak 风险」

  ▎ Target encoding 最大的风险是：如果对整个数据集做编码后再 split，test 数据的编码中就包含了 test 的 target
  ▎ 信息——这是严重的数据泄露。我的做法是：在 train split 上拟合所有编码器，然后对 test 数据用 train
  ▎ 的编码映射。对于训练集里没出现过的品牌/调香师，给一个平滑的全局均值作为默认值。

  ---
  第七轮：系统设计与工程实践

  Q15：「你的 Streamlit 应用里，8 个 persona 的 LLM 调用是并行还是串行？怎么实现的？」

  ▎ 是并行的，用 Python 的 ThreadPoolExecutor，8 个 worker 同时向 DeepSeek API 发请求。因为 LLM 调用是 IO-bound
  ▎ 的（等待网络响应），用多线程就可以充分利用等待时间。我要的不是 CPU 加速而是并行度——8 个请求大约 8-12
  ▎ 秒得到全部结果。如果不是并行的，就要 64-96 秒。

  Q16：「你的后端为什么用了 FastAPI + WebSocket，而不是直接用 Streamlit 跑？」

  ▎ Streamlit 是单线程、同步的，每次用户交互运行整个脚本。如果用 Streamlit 直接处理 8 个并行 LLM
  ▎ 调用，用户的页面会卡死到所有调用都回来。WebSocket 是流式协议，能向 Streamlit 逐步推送 persona
  ▎ 的评分结果——用户可以看到 8 个专家的评审一个个"到达"，体验更好。FastAPI 的异步支持也天然适合 LLM 调用的 IO 等待。

  Q17：「你的向量库是在内存里还是有磁盘持久化的？Streamlit 每次 rerun 会重新加载吗？」

  ▎ 是磁盘持久化的（ChromaDB PersistentClient），数据存在 chroma_fragrances/ 目录。Streamlit 中用 @st.cache_resource
  ▎ 修饰 get_vector_store()，确保向量库在应用生命周期内只加载一次，不会每次 rerun 重新加载——加载 ChromaDB 只需要数秒。

  ---
  第八轮：行为面试与软技能

  Q18：「你说多 Agent 协作是'失败'的，但在企业中往往需要多个 AI Agent 协作。你怎么区分'技术失败'和'不适合这个场景'？」

  ▎ 多 Agent 本身没问题，问题出在问题结构与架构选择的匹配度上。我这个任务本质是"检索 +
  ▎ 灌输"——关键信息来自外部数据库，不需要多步推理。多 Agent
  ▎ 适合"分解复杂推理"的场景（比如写代码、做研究、多步规划），但检索后再在 Agent 间传递信息只会引入压缩误差。

  追问：「那你怎么知道 LangGraph 多 Agent 在你的任务上不适合？如果 LangGraph 的作者坐在你对面，你怎么解释你的失败？」

  ▎ 我会指出具体的错误传播链路：Orchestrator 第一步把 accords+RAG 参考压缩成了简报，第二步 Top Specialist
  ▎ 读了简报（丧失了原始 RAG 参考的细节），第三步 Middle Specialist 读了前两者的输出（又丢了一轮信息）——经过 5 次 LLM
  ▎ 调用之后，原始 RAG 检索的细节基本没了。这不是 LangGraph 的代码问题，是架构范式与任务类型的不匹配。其实我期待
  ▎ LangGraph 的作者会同意——他们自己也在一直强调"不是每个任务都适合多 Agent"。

  Q19：「你最大的技术失败是什么？不是多 Agent 这个——说说别的。」

  这个问题一定要真诚：

  ▎ 最大失败是前期的评估方式。最初我纯粹用精确匹配
  ▎ F1，所有模型的表现看起来都很差。我有整整一周以为我的系统是坏掉的——直到我发现 "cedarwood" 和 "cedar"
  ▎ 被当作不同原料。这教给我一个重要的教训：评估指标体系的选择和模型架构一样重要。如果评估指标不能反映任务的实际质量标准
  ▎ ，所有优化都是在错误的坡度上爬山。

  Q20：「如果让你重做这个项目，你会做什么不同的选择？」

  ▎ 第一，从一开始就建立更好的评估体系。第二，会对 embedding 模型做领域适应——在香水文本上微调 MiniLM 或直接用 BGE-large
  ▎ 做 embedding，RAG 检索的精度会更高。第三，会用贝叶斯加权评分来做 RAG 检索的排序，而不是只看余弦相似度。第四，会加入
  ▎ IFRA 安全合规检查模块——这对于实际应用非常有价值，而且直接关联我在 DSM 了解的法规知识。
